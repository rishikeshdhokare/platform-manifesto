## 1. 🎯 Overview

"We gave everyone Copilot licenses" is not an AI adoption strategy. Without measurement, AI adoption stays anecdotal - some engineers love it, some ignore it, nobody knows if it is actually making the organization faster or better.

This document defines what to measure, how to measure it, and what to do with the results. The goal is not to maximize AI usage - it is to maximize engineering effectiveness, with AI as one of the levers.

---

## 2. 📈 Adoption Metrics

Adoption metrics answer: "Are engineers actually using AI tools?"

| Metric | Source | Target | Frequency |
|--------|--------|--------|-----------|
| **Active AI tool users** | License management dashboard (Copilot, Cursor) | > 80% of licensed engineers active weekly | Monthly |
| **AI suggestion acceptance rate** | Copilot/Cursor telemetry | 25-40% (lower suggests poor context; higher suggests uncritical acceptance) | Monthly |
| **Context file coverage** | CI/Backstage scorecard | 100% of repositories have AGENTS.md | Monthly |
| **Rules file freshness** | CI check | 100% of AGENTS.md updated within 90 days | Monthly |
| **AI-assisted PRs** | PR labels (`ai-assisted`) | Tracked, not targeted | Monthly |

### Reading Acceptance Rate

Acceptance rate is not a productivity metric - it is a context quality signal.

| Range | Interpretation | Action |
|-------|---------------|--------|
| < 15% | AI suggestions are irrelevant - context is missing or wrong | Audit AGENTS.md and rules files for the affected repositories |
| 15-25% | Typical for codebases without context engineering | Invest in repository context layer |
| 25-40% | Healthy range - AI suggestions are relevant, developer is selectively accepting | Maintain and refine context |
| > 50% | Suspiciously high - developer may be accepting without reviewing | Audit code quality in AI-heavy PRs |

## 3. 🚀 Effectiveness Metrics

Effectiveness metrics answer: "Is AI making us faster and better?"

| Metric | Measurement | Baseline | Target |
|--------|-------------|----------|--------|
| **PR cycle time** | Time from PR open to merge | Measure pre-AI baseline per team | 15-25% reduction |
| **Time to first commit** (new joiners) | Days from onboarding start to first PR merged | Measure pre-AI baseline | 30% reduction |
| **Boilerplate reduction** | Lines of scaffolding code generated per new service | Measure pre-AI baseline | 50% reduction |
| **Incident triage time** | Time from alert to root cause identification | Measure pre-AI baseline | 20% reduction |
| **Documentation coverage** | Percentage of services with up-to-date README, API docs, runbook | Current coverage | 90% coverage |

### Correlation, Not Causation

These metrics correlate with AI adoption but are influenced by many factors (team maturity, codebase complexity, product pressure). Do not attribute all improvement to AI tools. Use them as directional signals.

### What NOT to Measure

| Do Not Measure | Why |
|----------------|-----|
| Lines of code generated by AI | Incentivizes volume over quality |
| AI-generated vs human-written defect rates | Too many confounding variables; creates blame culture |
| Individual developer AI usage | Creates surveillance culture; adoption is a team concern, not an individual performance metric |

## 4. 🛡️ Quality Guardrails

AI adoption must not degrade code quality. Monitor these guardrails alongside adoption metrics.

| Guardrail | Metric | Threshold | Action if Breached |
|-----------|--------|-----------|-------------------|
| **Defect escape rate** | Bugs found in production per sprint | No increase from pre-AI baseline | Audit recent AI-assisted PRs for pattern |
| **Code review depth** | Average review comments per PR | No decrease from baseline | Retrain reviewers on AI-assisted PR review expectations |
| **Test quality** | Mutation testing kill rate | No decrease from baseline | Audit AI-generated tests for tautological or implementation-mirroring patterns |
| **Security findings** | SAST/DAST findings per sprint | No increase from baseline | Review AI-generated code in security-sensitive areas |
| **Technical debt** | SonarQube code smells, duplication | No increase from baseline | Review AI-generated code for unnecessary complexity |

## 5. 🪜 AI Engineering Maturity (L0-L4)

This is the **AI Engineering Maturity** track: context engineering, AI-assisted SDLC, adoption, and effectiveness of **developer-facing AI**. It is **not** the same as **ML Platform Maturity** ([Dimension 11](../08-program/01-maturity-model.md) in the program maturity model), which scores operational ML (serving, training and feature pipelines, production model observability, responsible-AI governance for deployed models). The two use similar level labels for familiarity, but they measure different capabilities. Review both in quarterly assessments where relevant.

Teams progress through maturity levels. Each level builds on the previous one. Skipping levels leads to superficial adoption.

| Level | Name | Characteristics | Context Engineering | SDLC Coverage |
|-------|------|----------------|--------------------|----|
| **L0** | Ad-hoc | Individual engineers use personal AI tools inconsistently | No AGENTS.md, no rules files | Code completion only |
| **L1** | Standardized | Team uses {Company}-licensed AI tools with guardrails from [AI Governance](../10-ai-ml-platform/02-ai-governance.md) | AGENTS.md exists with basic rules | Code completion + test generation |
| **L2** | Context-aware | Repository context layer is maintained; AI output aligns with team standards | AGENTS.md + cursor rules + ADRs in Git, reviewed quarterly | Development + testing + documentation |
| **L3** | Integrated | AI is embedded in CI/CD (automated review, canary analysis, doc freshness checks) | Full context stack maintained, freshness enforced in CI | All SDLC phases from [AI-Assisted SDLC](./02-ai-assisted-sdlc.md) |
| **L4** | Optimizing | Team measures AI effectiveness, runs experiments on context and prompting strategies, contributes to org-wide prompt libraries | Context quality is measured, feedback loop from output quality back to context improvement | Full SDLC + continuous improvement |

### Assessment

Teams self-assess quarterly as part of the broader [maturity model](../08-program/01-maturity-model.md) review. Where they ship or operate ML models, they also score **ML Platform Maturity** (Dimension 11). Where they use AI in engineering workflows, they score **AI Engineering Maturity** using the L0-L4 table in this section. The two scores are reported side by side when both apply; neither replaces the other.

### Progression Guide

| From | To | Key Actions | Typical Duration |
|------|-----|------------|-----------------|
| L0 | L1 | Provision {Company}-licensed tools, establish guardrails, run team training session | 2-4 weeks |
| L1 | L2 | Create AGENTS.md, add cursor rules for key patterns, move ADRs into Git | 1-2 sprints |
| L2 | L3 | Add AI review to CI, implement canary analysis, set up doc freshness checks | 1-2 quarters |
| L3 | L4 | Instrument metrics from this document, run A/B experiments on context strategies | Ongoing |

## 6. 📊 Reporting

### Team Dashboard

Every team's Backstage page includes an AI adoption panel showing:

- Active users / total licensed users
- Suggestion acceptance rate (with trend)
- Context health score (AGENTS.md presence, freshness, rules file count)
- PR cycle time trend (with AI adoption overlay)

### Organization Dashboard

Platform Engineering maintains an organization-level dashboard showing:

- Adoption by team
- Context health by team
- PR cycle time trends
- Quality guardrail status
- Maturity level distribution

### Quarterly Review

AI adoption metrics are reviewed quarterly alongside the [maturity model](../08-program/01-maturity-model.md) (including **ML Platform Maturity** for teams with production ML) and [engineering metrics](../08-program/04-engineering-metrics.md). Treat **AI Engineering Maturity** (this document) and **ML Platform Maturity** (Dimension 11) as separate lenses in the same review when both matter. The review covers:

1. Adoption trends - are teams progressing through AI Engineering Maturity levels?
2. Quality guardrails - are any teams showing degradation?
3. Context health - are rules files being maintained?
4. Investment recommendations - tooling, training, context infrastructure

## 7. 🔁 Continuous Improvement

### AI Champions Guild

Each team nominates an **AI champion** - an engineer responsible for:

- Maintaining the team's context files (AGENTS.md, cursor rules)
- Sharing effective prompt patterns within the team
- Contributing to the organization's prompt library
- Participating in the monthly AI Champions Guild meeting

The AI Champions Guild meets monthly to:

- Share patterns that worked across teams
- Identify common context gaps
- Propose updates to this document and the [AI-Assisted SDLC](./02-ai-assisted-sdlc.md)
- Review organization-level metrics

### Prompt Library

Effective prompt patterns are captured in a shared repository:

```
{company}/ai-prompt-library/
├── development/
│   ├── implement-from-interface.md
│   ├── generate-tests.md
│   └── migrate-pattern.md
├── review/
│   ├── security-review.md
│   └── performance-review.md
├── operations/
│   ├── incident-triage.md
│   └── pir-draft.md
└── documentation/
    ├── api-docs.md
    └── runbook-generation.md
```

Each prompt template includes: description, context requirements, example input, example output, and known limitations.

### Experimentation

Teams are encouraged to experiment with AI workflows:

- Try different context strategies and measure acceptance rate changes
- A/B test prompt patterns on similar tasks
- Document what works and what does not in the prompt library

> **Rule:** Experiments must not bypass quality guardrails. Trying a new prompting approach is fine; shipping unreviewed AI-generated code to test velocity is not.
